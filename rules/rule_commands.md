# 🎮 快捷命令体系

> **版本**: v1.3
> **创建时间**: 2026-03-12
> **创建原因**: 借鉴 SuperClaude Framework 命令系统，引入 `/gd:` 前缀快捷命令，提升操作效率
> **v1.1 更新**: 增强 `/gd:reflect` 命令，支持需求级+周期级两种内省模式
> **v1.2 更新**: 新增 `/gd:learn` 命令（借鉴ECC Instincts /learn），支持用户主动教授知识，自动写入知识库
> **v1.3 更新**: 新增 `/gd:audit` 命令（借鉴ECC /harness-audit），支持6维度规则体系全面健康度审计
> **触发方式**: 手动触发（当需要了解命令体系时加载）
> **核心原则**: 命令是"快捷入口"，不是"绕过流程"。命令只加速需求分类，后续仍走完整流水线

---

## 📌 命令体系总览

### 设计原则

| 原则 | 说明 |
|------|------|
| **不替代流程** | 命令是"快捷入口"，不是"绕过流程"。命令执行后仍走完整流水线 |
| **制作人有最终决定权** | 命令只是"建议"类型，制作人Agent仍有权纠正命令指定的类型 |
| **可选使用** | 用户可继续使用自然语言描述需求，命令只是额外的加速选项 |
| **前缀统一** | 所有命令统一使用 `/gd:` 前缀（GameDev 缩写） |
| **不跳过质量门禁** | 无论命令如何触发，质量门禁、对抗审查等强制环节不受影响 |
| **不跳过规模评估** | 命令跳过的只是"需求类型判断"步骤，规模评估和复合需求检查仍正常执行 |

### 命令格式

```
/gd:命令名 [参数]
```

- 以 `/gd:` 开头的用户输入识别为快捷命令
- 非 `/gd:` 开头的用户输入走标准自然语言分析流程（完全不受影响）
- 命令名不区分大小写

## 📋 命令分类与定义

### 一、需求类命令（进入开发流水线）

> 这类命令加速了制作人Agent的"需求类型判断"步骤，直接指定需求类型，但制作人仍执行规模评估和复合需求检查。

| 命令 | 格式 | 说明 | 对应类型 |
|------|------|------|---------|
| `/gd:new` | `/gd:new "需求描述"` | 新建需求，不指定类型（进入制作人标准分析流程） | 由制作人判断 |
| `/gd:feature` | `/gd:feature "功能描述"` | 直接按 FEATURE 类型启动 | FEATURE |
| `/gd:feature_ui` | `/gd:feature_ui "功能描述"` | 直接按 FEATURE_UI 类型启动 | FEATURE_UI |
| `/gd:bugfix` | `/gd:bugfix "问题描述"` | 直接按 BUGFIX 类型启动 | BUGFIX |
| `/gd:optimize` | `/gd:optimize "优化目标"` | 直接按 OPTIMIZE 类型启动 | OPTIMIZE |
| `/gd:config` | `/gd:config "配置说明"` | 直接按 CONFIG 类型启动 | CONFIG |
| `/gd:test` | `/gd:test "测试说明"` | 直接按 TEST 类型启动 | TEST |
| `/gd:doc` | `/gd:doc "文档说明"` | 直接按 DOC 类型启动 | DOC |
| `/gd:review` | `/gd:review "审查范围"` | 直接按 REVIEW 类型启动 | REVIEW |
| `/gd:research` | `/gd:research "调研主题"` | 直接按 RESEARCH 类型启动 | RESEARCH |

**需求类命令执行逻辑**：

```
用户输入 /gd:bugfix "背包界面打开报空"
    │
    ├── 1. 提取命令中的类型信息（bugfix → BUGFIX）
    ├── 2. 提取参数中的需求描述（"背包界面打开报空"）
    ├── 3. 制作人Agent跳过"关键词识别"步骤（1.3），直接使用命令指定的类型
    ├── 4. 制作人Agent仍执行以下步骤（不可跳过）：
    │   ├── 1.3b 产出物→Agent归属检查
    │   ├── 1.4 复合需求三重检查
    │   ├── 1.5 模糊需求处理（如命令参数描述不清）
    │   └── 1.6 规模评估
    ├── 5. 如果制作人判断命令类型不准确，有权覆盖并说明原因：
    │   └── 输出: "⚠️ 命令指定为BUGFIX，但经分析实际为FEATURE，已自动调整"
    └── 6. 后续按标准流水线流程走
```

**`/gd:new` 特殊说明**：
- `/gd:new` 不指定类型，与直接用自然语言描述需求的效果相同
- 唯一区别是明确告知系统"这是一个新需求"，便于日志记录

### 二、管理类命令（不进入开发流水线）

> 这类命令直接执行查询/管理操作，不启动流水线，执行完毕后等待用户下一步指令。

| 命令 | 格式 | 说明 | 执行操作 |
|------|------|------|---------|
| `/gd:status` | `/gd:status` | 查看所有需求状态 | 读取并展示 `_ProjectManagement/需求池.md` |
| `/gd:progress` | `/gd:progress` | 查看当前进度看板 | 读取并展示 `_ProjectManagement/进度看板.md` |
| `/gd:resume` | `/gd:resume [REQ-ID]` | 恢复指定需求的上下文 | 执行上下文恢复协议 |
| `/gd:resume` | `/gd:resume` | 恢复最近的进行中需求 | 查找最近的进行中需求，执行恢复协议 |
| `/gd:plan` | `/gd:plan` | 查看全局策划案摘要 | 读取并展示 `全局策划案.md` 摘要 |
| `/gd:tech` | `/gd:tech` | 查看全局技术文档摘要 | 读取并展示 `全局技术文档.md` 摘要 |
| `/gd:kb` | `/gd:kb` | 查看知识库摘要 | 读取并展示所有知识库文件的目录摘要 |
| `/gd:history` | `/gd:history [REQ-ID]` | 查看指定需求的完整历史 | 读取需求文档头部的frontmatter，展示完整时间线 |
| `/gd:stats` | `/gd:stats` | 查看时长统计数据 | 读取并展示 `_ProjectManagement/时长统计.md` |
| `/gd:issues` | `/gd:issues` | 查看问题日志 | 读取并展示 `_ProjectManagement/问题日志.md` |

**管理类命令执行逻辑**：

```
用户输入 /gd:status
    │
    ├── 1. 识别为管理类命令
    ├── 2. 不启动开发流水线
    ├── 3. 不启动制作人Agent
    ├── 4. 直接执行对应的读取/展示操作
    └── 5. 完成后等待用户下一步指令
```

**`/gd:resume` 执行逻辑**：

```
用户输入 /gd:resume REQ-003
    │
    ├── 1. 读取 需求池.md → 查找 REQ-003
    ├── 2. 读取该需求文档头部的 frontmatter
    ├── 3. 确认 current_agent 和 current_step
    ├── 4. 加载该 Agent 对应步骤的规范文件
    ├── 5. 输出: "🔄 恢复执行 REQ-003，当前在 XXAgent XXX阶段"
    └── 6. 继续执行流水线

用户输入 /gd:resume （不带参数）
    │
    ├── 1. 读取 需求池.md → 查找 status=🔄进行中 的最新需求
    ├── 2. 如果有多个进行中需求，选择最近更新的那个
    ├── 3. 按上述恢复协议执行
    └── 4. 如果没有进行中需求，输出: "📭 当前没有进行中的需求"
```

### 三、迭代类命令（触发反思/迭代流程）

> 这类命令触发反思或迭代改进流程，不属于常规开发流水线。

| 命令 | 格式 | 说明 | 执行操作 |
|------|------|------|----------|
| `/gd:reflect` | `/gd:reflect` | 触发**周期级**结构化内省 | 统计最近N个需求的指标、识别模式、评估规则健康度 |
| `/gd:reflect` | `/gd:reflect REQ-XXX` | 触发**指定需求**的内省报告 | 对指定需求执行需求级结构化内省（等同step-D2） |
| `/gd:iterate` | `/gd:iterate "问题描述"` | 触发迭代改进流程 | 对指定问题执行5 Why分析 → 定位规则 → 更新规则 |
| `/gd:audit` | `/gd:audit` | **全面审计规则体系健康度** | 6维度检查 → 输出异常项 → 生成行动清单 |
| `/gd:audit` | `/gd:audit [维度]` | 审计**指定维度** | 只审计指定维度（规则/Agent/知识库/防护/指标/文档） |

**`/gd:reflect` 执行逻辑（借鉴SuperClaude `/sc:reflect`）**：

```
用户输入 /gd:reflect
    │
    ├── 1. 识别为迭代类命令
    ├── 2. 判断是否带参数
    │   ├── 无参数 → 周期级内省
    │   │   ├── 读取需求池，统计最近已完成的需求
    │   │   ├── 汇总各需求的内省报告数据
    │   │   ├── 计算核心指标均值和趋势
    │   │   ├── 识别反复出现的成功/失败模式
    │   │   ├── 评估规则健康度和知识库增长
    │   │   ├── 使用 `02_策划Agent/templates/周期内省模板.md` 生成报告
    │   │   └── 输出行动清单
    │   │
    │   └── 带 REQ-ID → 需求级内省
    │       ├── 读取指定需求的文档和frontmatter
    │       ├── 按 `02_策划Agent/step-D2_需求内省.md` 执行
    │       └── 生成需求级内省报告
    │
    └── 3. 完成后等待用户下一步指令

用户输入 /gd:reflect REQ-003
    │
    ├── 1. 识别为指定需求内省
    ├── 2. 读取 REQ-003 的需求文档和frontmatter
    ├── 3. 按 step-D2 的执行清单进行结构化内省
    └── 4. 生成并输出内省报告
```

**`/gd:iterate` 执行逻辑**：

```
用户输入 /gd:iterate "制作人经常把FEATURE误判为BUGFIX"
    │
    ├── 1. 识别为迭代类命令
    ├── 2. 执行5 Why根因分析
    ├── 3. 定位需更新的规则文件
    ├── 4. 立即执行更新
    └── 5. 记录迭代日志
```

**`/gd:audit` 执行逻辑（借鉴ECC /harness-audit）**：

```
用户输入 /gd:audit
    │
    ├── 1. 识别为迭代类命令
    ├── 2. 确认审计范围：全量审计（6维度）
    ├── 3. 设置计时器（10分钟上限）
    ├── 4. 逐维度执行检查：
    │   ├── 📏 规则完整性：交叉验证 rule.md + 子规则文件
    │   ├── 🤖 Agent一致性：对比 rule.md Agent列表 + 各入口文件
    │   ├── 📚 知识库健康度：检查格式、活跃度、废弃条目
    │   ├── 🛡️ 防护规则有效性：检查 rule_guards.md + 最近失败模式
    │   ├── 📊 指标达标：计算返工率/门禁通过率等核心指标
    │   └── 📋 文档同步性：抽查关键引用路径和版本号
    │
    ├── 5. 汇总异常项，按严重度分级（🔴🟡🟢）
    ├── 6. 生成行动清单（按优先级排序）
    ├── 7. 输出审计报告（只展示异常项）
    └── 8. 询问用户是否立即执行修复

用户输入 /gd:audit 知识库
    │
    ├── 1. 识别为部分审计
    ├── 2. 只执行维度3（📚 知识库健康度）检查
    ├── 3. 输出该维度的审计结果
    └── 4. 生成该维度的行动清单
```

**`/gd:audit` 可指定的维度名称**：

| 参数 | 审计维度 | 检查重点 |
|------|---------|----------|
| `规则` | 📏 规则完整性 | 版本号、索引、不可变规则、分层体系 |
| `Agent` | 🤖 Agent一致性 | 入口文件、职责表、权限声明、步骤索引 |
| `知识库` | 📚 知识库健康度 | 格式、活跃度、废弃条目、置信度分布 |
| `防护` | 🛡️ 防护规则有效性 | 规则列表、高频问题、触发记录 |
| `指标` | 📊 指标达标 | 返工率、门禁通过率、审查发现率等 |
| `文档` | 📋 文档同步性 | 引用路径、模板版本、命令列表 |

> **详细的审计检查清单、输出格式、与内省的协同关系**见 `rule_iteration.md` 中的"🔬 定期审计"章节。

> **`/gd:reflect` 与需求内省(step-D2)的关系**：
> - 每个需求完成后自动执行 step-D2（需求级内省）
> - `/gd:reflect` 无参数时触发**周期级内省**（跨需求统计分析）
> - `/gd:reflect REQ-XXX` 可对指定需求补做或重做内省

## 🔍 命令识别规则

### 识别优先级

```
用户输入
    │
    ├── 以 /gd: 开头？
    │   ├── 是 → 识别为快捷命令
    │   │   ├── 匹配到已定义的命令名？
    │   │   │   ├── 是 → 按命令类型执行
    │   │   │   └── 否 → 输出: "❌ 未识别的命令：/gd:xxx，请检查命令名"
    │   │   │        并列出可用命令列表
    │   │   └── 判断命令类别（需求类/管理类/迭代类）→ 按对应逻辑执行
    │   │
    │   └── 否 → 走标准自然语言分析流程（完全不受命令体系影响）
    │
    └── 同时匹配到旧版快速启动命令？（如"开发 xxx"、"查看进度"）
        └── 旧版命令继续兼容，与 /gd: 命令并行生效
```

### 命令参数规则

| 规则 | 说明 |
|------|------|
| 参数可用引号包裹 | `/gd:bugfix "背包界面报空"` |
| 参数也可不用引号 | `/gd:bugfix 背包界面报空` |
| 无参数命令直接执行 | `/gd:status` |
| 可选参数缺省时使用默认行为 | `/gd:resume` → 恢复最近的进行中需求 |

### 命令与旧版快速启动命令的兼容关系

| 旧版命令 | 新版 `/gd:` 命令 | 兼容方式 |
|---------|-----------------|---------|
| `开发 {功能描述}` | `/gd:feature "功能描述"` | 两者均可使用 |
| `查看进度` | `/gd:progress` | 两者均可使用 |
| `查看需求池` | `/gd:status` | 两者均可使用 |
| `查看统计` | `/gd:stats` | 两者均可使用 |
| `查看问题` | `/gd:issues` | 两者均可使用 |
| `查看全局策划` | `/gd:plan` | 两者均可使用 |
| `查看全局技术` | `/gd:tech` | 两者均可使用 |
| `继续上次的需求` | `/gd:resume` | 两者均可使用 |
| `恢复 REQ-003` | `/gd:resume REQ-003` | 两者均可使用 |

> **兼容策略**：旧版自然语言命令全部保留，新版 `/gd:` 命令作为额外的标准化选项。用户可按习惯自由选择。

## ⚠️ 注意事项

1. **命令不替代流程**：需求类命令只加速了制作人的"类型判断"步骤，后续流程不受影响
2. **制作人有最终决定权**：命令指定的类型只是"建议"，制作人有权覆盖
3. **质量门禁不可跳过**：无论通过命令还是自然语言触发，3级质量门禁、对抗审查等强制环节始终执行
4. **规模评估不可跳过**：命令触发后仍需执行规模评估（XS/S/M/L/XL）
5. **复合需求检查不可跳过**：命令触发后仍需执行产出物归属检查和三重检查法
6. **命令参数即需求原文**：命令参数中的描述将作为"用户原始需求"记录
