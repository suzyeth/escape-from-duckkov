# Step 02: 进度监控 [MODE: REVIEW]

## 📍 当前位置
持续监控 | 贯穿整个需求生命周期 | 🔍 审查模式

## 🎯 本步骤目标
实时监控每个阶段的时长，发现时间风险及时通知用户，但绝不以牺牲质量为代价压缩时间。

## 📋 监控规则

### 每个阶段开始时
- 记录开始时间
- 更新进度看板
- **更新需求状态追踪frontmatter**：`current_agent`、`current_step`、新增`steps_completed`条目
- **检查需求规模**：如果是L/XL规模，确认主从架构已启用

### 每个阶段结束时
- 记录结束时间，计算时长
- 更新进度看板和需求池
- **更新需求状态追踪frontmatter**：对应条目的`end_time`、`duration_min`、`status=done`、更新`progress`百分比
- **L/XL规模特别注意**：子任务的时长需要单独记录，主任务汇总总时长

### 超时处理流程

```
阶段时长 > 预期时长 × 1.5
    ↓
🟡 通知用户：存在时间风险，但当前阶段仍需保质完成
    ↓
记录到问题日志
    ↓
通知对应 Agent 报告进度和阻塞原因（不要求加速赶工）
    ↓
阶段时长 > 预期时长 × 2
    ↓
🔴 升级为严重超时
    ↓
通知用户：时间风险较大，由用户决定是否调整范围
```

## 🚨 超时通知格式

```markdown
## ⚠️ 阶段时间风险通知

| 属性 | 值 |
|------|-----|
| 需求ID | REQ-XXX |
| 当前阶段 | XXX |
| 负责Agent | XXX Agent |
| 预期时长 | XX分钟 |
| 实际时长 | XX分钟 |
| 超时程度 | 超出 XX% |

### 请 {Agent名称} 报告：
1. 当前进度如何？
2. 遇到什么阻塞？
3. 预计还需多久？
4. 是否需要协助？
```

## ⚠️ 本步骤常见陷阱
- 预估过于乐观，低估需求复杂度
- 忽略跳过阶段的时长计算
- 问题不分类，难以分析
- ❌ 为赶时间要求Agent降低质量（应通知用户由用户决策）
