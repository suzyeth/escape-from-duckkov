使用以下模板输出（或加载 `templates/需求分析输出模板.md`）：

┌─────────────────────────────────────────────────────┐

### 2.2 确认流转路径

| 类型代码 | 流转路径 |
|----------|----------|
| `FEATURE` | 项目管理 → 策划 → 主程 → 程序 → QA |
| `OPTIMIZE` | 项目管理 → 主程 → 程序 → QA |
| `BUGFIX` | 项目管理 → 程序 → QA |
| `TEST` | 项目管理 → QA |
| `DOC` | 项目管理 → 主程 |
| `REVIEW` | 项目管理 → 主程 → 程序 |
| `CONFIG` | 项目管理 → 程序 → QA |
| `RESEARCH` | 策划（跳过项目管理） |

### 2.3 🚨 流转前越权检查（强制步骤）

**在执行流转之前，必须逐一验证以下内容：**

- [ ] 列出本次需求的所有预期产出物
- [ ] 逐一确认每个产出物将由流转路径中的哪个Agent负责
- [ ] **🔴 如果发现某个产出物不属于流转路径中任何Agent的职责 → 停止！重新检查需求分类**
- [ ] **🔴 如果发现需要两条不同的流转路径才能覆盖所有产出物 → 停止！回到step-01重新拆分为复合需求**

> **历史教训**: "全局策划案+全局技术文档" 被错误归类为单一DOC类型流转给主程，导致主程越权产出了策划案。
> 正确做法：发现两个产出物分属策划Agent和主程Agent → 回到step-01拆分为两个子需求。

### 2.4 执行流转

- [ ] 输出分析结果框
- [ ] 对于RESEARCH类型：直接流转给策划Agent
- [ ] 对于其他类型：流转给项目管理Agent

### 2.5 🔄 强制自我反思与迭代触发（每次执行后必须执行）

**🚨 这不是可选步骤，每次流转前必须完成以下反思：**

> 📌 通用反思清单和迭代四步法定义在 `rule.md` 的"强制自我反思与迭代触发原则"中，所有Agent通用。
> 以下是制作人Agent在通用反思基础上的**额外专属检查项**：

```
□ [通用] 本阶段产出物是否完整且准确？
□ [通用] 产出物归属是否正确？（是否有越权产出不属于自己的内容？）
□ [通用] 本阶段是否有流程偏差？（对照step文件逐项验证）
□ [通用] 本阶段是否有可改进的做法？
□ [通用] 🔴 本次执行中是否犯了错误？（包括被用户纠正的错误）
─── 以下为制作人专属检查项 ───
□ [专属] 需求分类是否准确？（对照1.3b产出物归属表逐一验证）
□ [专属] 规模评估是否合理？（有无低估或高估）
□ [专属] 🔍 三级识别体系是否正确执行？（一级→二级→上下文→语义，逐级递进）
□ [专属] 🔍 关键词识别是否有遗漏？需要在一级/二级关键词表中添加新词？
□ [专属] 🔍 上下文线索是否被充分利用？（文件名、错误日志、用户语气等）
□ [专属] 流转路径是否合理？是否存在越权风险？
□ [专属] 是否有新的需求类型无法归类？
□ [专属] `/gd:` 命令识别是否正确？命令覆盖是否合理？
□ [专属] 类型置信度标注是否与判断来源匹配？（命令→🟢、一级关键词→🟢、二级+上下文→🟡、语义→🔴）
□ [专属] 🔍 判断依据是否具体？（不能只写"关键词命中"，需说明哪个关键词+什么上下文）
```

**🔄 迭代触发规则：** 参照 `rule.md` 中的通用迭代四步法执行。

## ⚠️ 注意事项

1. **不确定时询问**: 如果需求模糊，主动询问用户
2. **复合需求拆分**: 如果一个需求包含多种类型，拆分成多个子需求
3. **保守判断**: 不确定是否需要策划时，走完整流程
4. **记录决策**: 所有判断都要记录依据，便于追溯
