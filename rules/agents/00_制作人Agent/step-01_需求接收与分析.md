### 1.2 参考历史数据

### 1.3 增强版关键词识别与智能分类（🔍 借鉴SuperClaude自动激活机制）

> ⏭️ **命令快速路径**：如果在 1.0 中已通过 `/gd:` 需求类命令指定了类型，本步骤跳过，直接进入 1.3b。
> 但如果命令为 `/gd:new`（未指定类型），仍需执行本步骤。

**采用三级识别体系**：一级关键词 → 二级关键词+上下文 → 语义分析，逐级递进，确保分类准确。

#### 1.3.1 一级关键词匹配（高置信度 🟢）

> 直接命中 → 直接确认类型，置信度标记为 🟢高。

| 类型代码 | 一级关键词（直接命中） | 示例 |
|----------|----------------------|------|
| `FEATURE` | 开发、实现、新增功能、添加、创建、做一个、制作 | "开发一个仓库系统" |
| `OPTIMIZE` | 优化、重构、性能、提速、减少GC、改进代码 | "优化仓库系统的性能" |
| `BUGFIX` | 修复、bug、问题、报错、崩溃、异常、空引用、NullReference | "修复背包无法打开的bug" |
| `TEST` | 测试、测试用例、覆盖率、单元测试、补充测试 | "给仓库系统增加测试用例" |
| `DOC` | 检查文档、更新文档、**技术文档**、全局技术文档 | "检查技术文档是否完整" |
| `REVIEW` | 审查、review、检查代码、代码质量 | "审查仓库系统的代码" |
| `CONFIG` | 配置、参数、调整、修改数值 | "调整背包容量上限" |
| `RESEARCH` | 调研、分析、竞品、待做清单、功能规划、方向、**策划案**、全局策划案 | "分析市面上互动影游的功能" |

> ⚠️ **一级关键词冲突时**：当需求同时包含多种类型的一级关键词时（如"策划案"和"技术文档"），不要强行选择一种类型，而应进入 **1.4 复合需求检查** 进行拆分。

#### 1.3.2 二级关键词匹配（中置信度 🟡）

> 一级关键词未命中时，检查二级关键词。二级关键词需结合上下文线索才能确认类型。

| 二级关键词 | 可能类型 | 判断依据 |
|-----------|---------|----------|
| UI、界面、面板、按钮、弹窗 | FEATURE_UI / BUGFIX | 如果是新增UI → FEATURE_UI；修复UI → BUGFIX |
| 网络、联机、多人、同步 | FEATURE | 大概率涉及新功能 |
| 存档、保存、加载、持久化 | FEATURE / BUGFIX | 根据"新增"还是"修复"判断 |
| 排序、筛选、过滤 | FEATURE / BUGFIX / CONFIG | 新增排序→FEATURE；排序不对→BUGFIX；调整排序规则→CONFIG |
| 动画、特效、表现 | FEATURE / BUGFIX | 新增表现→FEATURE；表现异常→BUGFIX |
| 数据、表格、配置表 | CONFIG / FEATURE | 调整已有数据→CONFIG；新增数据模型→FEATURE |
| 改一下、调一下、换成 | CONFIG / BUGFIX | 简单参数调整→CONFIG；修正错误值→BUGFIX |

**二级关键词判断流程**：
```
二级关键词命中
    │
    ├── 结合上下文线索确认（见1.3.3）
    │   ├── 上下文支持 → 确认类型，置信度 🟡中
    │   └── 上下文不足/矛盾 → 进入语义分析（1.3.4）
    │
    └── 判断依据记录："二级关键词'{关键词}'命中 + 上下文'{线索}'确认为{类型}"
```

#### 1.3.3 上下文感知规则（辅助二级关键词判断）

> 🔍 **上下文感知**：不仅看关键词，还分析需求的上下文信息来辅助判断。

| 上下文线索 | 推断逻辑 | 适用场景 |
|-----------|---------|----------|
| 用户提到了**具体的.cs文件名** | 大概率是 BUGFIX 或 OPTIMIZE | "ItemListPanel.cs报空"→BUGFIX |
| 用户提到了**具体的错误日志/堆栈** | 确认 BUGFIX | "NullReferenceException at line 42"→BUGFIX |
| 用户描述了**一个新的玩法/机制** | 确认 FEATURE | "做一个自动战斗的新玩法"→FEATURE |
| 用户提到了**"打开XX面板"/"点击XX按钮"** | 如果现有功能异常→BUGFIX；如果新增→FEATURE_UI | 区分"打开面板报错"和"新增一个面板" |
| 用户提到了**具体的数值** | 大概率是 CONFIG | "背包上限从20改成50"→CONFIG |
| 用户使用了**否定/抱怨语气** | 倾向 BUGFIX | "XXX不能用了/XXX坏了"→BUGFIX |
| 用户使用了**期望/想要语气** | 倾向 FEATURE | "我想要一个XXX/能不能做个XXX"→FEATURE |
| 需求描述**涉及多个模块协作** | 规模可能偏大（L/XL） | 辅助 1.6 规模评估 |
| **需求池中有相关的已完成需求** | 参考历史类型做辅助判断 | 如果之前做过类似需求，参考其类型 |

**上下文感知判断原则**：
- 📌 上下文线索是**辅助判断**，不是独立判断依据
- 📌 当上下文线索与关键词矛盾时，以上下文线索为准（因为上下文更贴近用户真实意图）
- 📌 当多条上下文线索互相矛盾时，进入语义分析（1.3.4）或主动询问用户

#### 1.3.4 语义分析（低置信度 🔴）

> 当一级、二级关键词均未命中，且上下文线索不足时，进行纯语义理解。

- [ ] 分析需求的核心意图（用户**真正想做什么**）
- [ ] 分析需求的预期产出物（用户期望**得到什么**）
- [ ] 根据意图和产出物推断最可能的类型
- [ ] 置信度标记为 🔴低
- [ ] 判断依据记录："纯语义分析，{具体分析过程}"

> ⚠️ 当置信度为 🔴低 时，在输出中增加提示："⚠️ 类型判断置信度较低，如分类不准确请指正"

#### 1.3.5 三级识别体系总览流程

```
用户输入需求（自然语言模式）
    │
    ├── 一级关键词命中？
    │   ├── 是（单一类型）→ 确认类型，置信度 🟢高
    │   ├── 是（多类型冲突）→ 进入 1.4 复合需求检查
    │   └── 否 → 继续二级检查 ↓
    │
    ├── 二级关键词命中？
    │   ├── 是 → 结合上下文线索判断（1.3.3）
    │   │   ├── 上下文确认 → 确认类型，置信度 🟡中
    │   │   └── 上下文不足/矛盾 → 继续语义分析 ↓
    │   └── 否 → 继续语义分析 ↓
    │
    └── 语义分析（1.3.4）
        ├── 分析成功 → 确认类型，置信度 🔴低
        └── 无法判断 → 进入 1.5 模糊需求处理（主动询问用户）
```

### 1.3b 产出物→Agent归属检查（🚨 强制步骤）

**在确定类型之前，先检查需求中的每个产出物分别归属哪个Agent**：

| 产出物关键词 | 归属Agent | 对应需求类型 |
|-------------|-----------|-------------|
| 策划案、全局策划案、玩法设计、功能设计 | **策划Agent** | RESEARCH / FEATURE |
| 技术文档、全局技术文档、架构文档 | **主程Agent** | DOC |
| 代码、脚本、实现 | **程序Agent** | FEATURE / BUGFIX |
| 测试报告、测试用例 | **QA Agent** | TEST |
| UX设计、交互设计、界面布局 | **UX Agent** | FEATURE_UI |

- [ ] 列出需求中所有预期产出物
- [ ] 逐一检查每个产出物归属哪个Agent
- [ ] **如果产出物归属 ≥ 2个不同Agent → 这是复合需求，必须拆分**

### 1.4 复合需求检查（🚨 三重检查法）

**必须依次执行以下三种检查方法，任一触发即为复合需求：**

**检查法1: 关键词多类型检查**
- [ ] 需求文本中是否同时包含多种类型的关键词？（如同时含"策划案"+"技术文档"）

**检查法2: 产出物多Agent检查（最重要！）**
- [ ] 需求的多个产出物是否分别归属不同的Agent？
- [ ] 如果产出物A由策划Agent负责，产出物B由主程Agent负责 → **必须拆分**
- [ ] ⚠️ **即使产出物都叫"文档"，只要归属不同Agent，就必须拆分**

**检查法3: 流转路径冲突检查**
- [ ] 如果不拆分，能否用一条流转路径覆盖所有产出物？
- [ ] 如果需要两条不同的流转路径 → **必须拆分**

**拆分后处理：**
- [ ] 每个子需求独立分类和评估规模
- [ ] 确定子需求执行顺序（有依赖关系时先执行被依赖项）
- [ ] 记录拆分依据

> **历史教训**: 「全局策划案+全局技术文档」因为都是"文档"被误判为单一DOC类型，实际上策划案归属策划Agent(RESEARCH)、技术文档归属主程Agent(DOC)，必须拆分。

### 1.5 模糊需求处理
- [ ] 如果需求模糊无法分类，**主动询问用户**
- [ ] 不确定是否需要策划时，**保守判断，走完整流程**

### 1.6 📏 规模评估（新增）

在确定需求类型后，评估需求规模：

| 规模 | 判断标准 | 流程模式 |
|------|---------|--------|
| **中小 (XS/S/M)** | 300行以内代码，不涉及多模块协作 | 标准模式：完整流程 |
| **大 (L)** | 多模块协作，300行以上代码 | 主从模式：拆分子任务 |
| **超大 (XL)** | 新系统/跨系统改造 | 深度模式：增加评审环节 |

- [ ] 评估需求涉及的模块数量
- [ ] 评估预估代码量（基于经验和类似需求历史）
- [ ] 评估是否涉及新系统/跨系统改造
- [ ] 确定规模等级（XS/S/M 或 L 或 XL）
- [ ] 确定对应的流程模式

**规模评估辅助问题**：
1. 这个需求涉及几个独立模块？（≥3 → 可能是L）
2. 预估代码量超过300行吗？（是 → 可能是L）
3. 是否需要从零开始构建新系统？（是 → 可能是XL）
4. 是否涉及多个现有系统的深度改造？（是 → 可能是XL）

> **注意**：规模评估仅适用于 FEATURE/FEATURE_UI/OPTIMIZE 类型。BUGFIX/CONFIG/TEST/DOC/REVIEW/RESEARCH 默认为中小规模。

### 1.7 RESEARCH特殊说明
如果判断为RESEARCH类型：
- 不走项目管理（不分配需求ID、不计时）
- 产出文档位置：`.GameDev/_Research/` 目录
- 不更新全局策划案（仅供用户决策参考）

## ⚠️ 常见陷阱

| 陷阱 | 后果 | 避免方法 |
|------|------|----------|
| **过度简化** - 把FEATURE误判为BUGFIX | 跳过策划导致返工 | 仔细分析需求本质 |
| **忽略UI需求** - 未识别需求涉及UI | 跳过UX设计 | 检查二级关键词中的UI相关词 |
| **模糊需求直接分类** | 分类错误 | 不确定时主动询问 |
| **规模低估** - 将L规模判断为XS/S/M | 不拆分导致质量下降 | 用辅助问题确认模块数和代码量 |
| **规模高估** - 将XS/S/M规模判断为L | 不必要的拆分增加开销 | 保守判断，不确定时选中小规模 |
| **🔴 产出物合并** - 多个不同Agent负责的产出物被当成一个需求 | 越权：某Agent产出了不属于自己职责的文档 | **执行1.3b产出物归属检查** |
| **🔴 关键词表面化** - 只看"文档"就归类DOC，忽略"策划案"归属策划Agent | 策划类产出被错误分配给主程 | **先查产出物归属，再定类型** |
| **🔴 只看关键词不看上下文** - "修复排序"不一定是BUGFIX，可能是CONFIG | 类型误判 | **使用三级识别体系，结合上下文线索** |
| **🔴 忽略用户语气** - 用户说"XXX能不能做一个"，虽无明确关键词但意图是FEATURE | 类型遗漏 | **启用1.3.3上下文感知规则** |
