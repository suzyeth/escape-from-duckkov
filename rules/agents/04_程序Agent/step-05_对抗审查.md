# Step 05: 自我对抗审查 ⚔️ [MODE: REVIEW]

## 📍 当前位置
步骤 5/5 | 前置：质量门禁已通过 | 后续：流转QA Agent | 🔍 审查模式 — ⚠️ 切换为"对抗者"心态

## 🎯 本步骤目标
**切换为"对抗者"心态**，主动找出自己代码中的问题。
不是在验证代码是否正确，而是在**寻找代码中的缺陷**。

> ⚠️ 你现在是一个**严格的代码审查者**，不是代码的编写者。
> 你比写这些代码的"开发者自己"更挑剔、更仔细。

---

## ⚡ 审查铁律

1. **每次审查至少发现 3 个具体问题** — 禁止"看起来不错"的敷衍结论
2. **如果找不到3个问题，说明你不够仔细** — 重新审查，换个角度
3. **问题必须具体** — 必须指出文件名、行号/位置、问题描述
4. **问题数量上限10个** — 防止无限审查，聚焦最重要的问题
5. **审查时间控制在10分钟内** — 超时则只关注🔴严重问题

---

## 📋 审查攻击计划

### 攻击1: 验收标准覆盖审计（最重要）

> 逐条对照策划案的AC-XXX，验证代码是否**真正**覆盖

```
对每条 AC-XXX:
1. 找到对应的实现代码位置
2. 验证正常流程是否覆盖
3. 验证边界条件是否处理
4. 验证异常情况是否处理
5. 如果找不到对应代码 → 🔴 严重问题
```

### 攻击2: 代码质量深潜

> 以"找茬"心态逐文件审查

| 攻击方向 | 具体检查 |
|---------|---------|
| **null安全** | 所有外部引用是否都有null检查？尤其是GetComponent、Find、序列化字段引用 |
| **内存安全** | Update/FixedUpdate中是否有GC分配？（new、string拼接、LINQ、闭包） |
| **线程安全** | 协程中是否有竞态条件？异步操作结果是否有过期检查？ |
| **命名规范** | 是否符合项目命名规范？有无拼写错误、含义不清的命名？ |
| **单一职责** | 有无类承担过多职责？单个方法是否超过50行？ |
| **硬编码** | 有无魔法数字或硬编码字符串？应该提取为常量的值？ |
| **错误处理** | 是否有空的catch块？是否有被吞掉的异常？ |
| **边界情况** | 除以零？空集合操作？索引越界？整数溢出？ |

### 攻击3: Unity特定审查

| 攻击方向 | 具体检查 |
|---------|---------|
| **性能陷阱** | Update中Find/GetComponent？频繁Instantiate/Destroy？ |
| **生命周期** | Awake/Start/OnEnable/OnDisable/OnDestroy使用是否正确？ |
| **序列化** | SerializeField默认值合理？private字段是否误标了[SerializeField]？ |
| **协程** | StartCoroutine有对应StopCoroutine？OnDestroy中是否停止协程？ |
| **事件** | 事件订阅是否在OnDestroy中取消？是否有重复订阅风险？ |

### 攻击4: 架构层面审查

| 攻击方向 | 具体检查 |
|---------|---------|
| **耦合度** | 类之间是否过度耦合？是否可以通过接口解耦？ |
| **扩展性** | 如果需求变更，这个设计是否容易扩展？ |
| **一致性** | 新代码的风格是否与项目现有代码一致？ |
| **重复代码** | 是否有复制粘贴的重复代码？是否可以提取公共方法？ |

## 📊 审查报告格式

> 必须按照以下格式输出审查报告

```markdown
## ⚔️ 对抗审查报告

**审查范围**: [列出审查的文件]
**审查时间**: [X分钟]
**发现问题数**: [N个]

### 问题清单

| # | 严重程度 | 文件 | 位置 | 问题描述 | 修复建议 |
|---|---------|------|------|---------|---------|
| 1 | 🔴 严重 | xxx.cs | L45 | [具体问题] | [修复建议] |
| 2 | 🔴 严重 | xxx.cs | L88 | [具体问题] | [修复建议] |
| 3 | 🟡 警告 | xxx.cs | L12 | [具体问题] | [修复建议] |
| 4 | 🟢 建议 | xxx.cs | L30 | [具体问题] | [修复建议] |

### 问题统计
- 🔴 严重: X个（必须修复）
- 🟡 警告: X个（建议修复）
- 🟢 建议: X个（可选修复）
```

---

## 🔧 修复规则

| 严重程度 | 修复要求 | 说明 |
|---------|---------|------|
| 🔴 **严重** | **必须修复** | 不修复将导致功能错误、崩溃、安全问题 |
| 🟡 **警告** | **建议修复**，可标注为TODO | 不修复可能导致性能问题、维护困难 |
| 🟢 **建议** | **可选修复** | 代码风格、最佳实践建议 |

### 修复流程

```
审查报告输出后:
├── 有🔴严重问题?
│   ├── 是 → 立即修复所有🔴问题 → 修复后重新审查受影响的代码
│   └── 否 → 选择性修复🟡问题 → 标注🟢为TODO
└── 修复完成 → 更新审查报告 → 流转QA
```

### 重新审查规则
- 修复🔴严重问题后，**只审查修复相关的代码**（不需要全量重新审查）
- 如果修复引入了新问题，需要再次修复
- **最多进行2轮修复审查**，超过2轮则直接流转QA（避免无限循环）

---

## ⚠️ 本步骤常见陷阱

1. **敷衍审查**：只列3个无关紧要的建议就通过 → 必须有实质性发现
2. **过度审查**：列了20个问题，大部分是鸡毛蒜皮 → 上限10个，聚焦重要问题
3. **审查后不修复**：发现🔴严重问题但不修复就流转 → 严重问题必须修复
4. **修复引入新Bug**：修复时不够仔细 → 修复后需要验证受影响代码
5. **忘记切换心态**：以编写者心态审查，自然觉得没问题 → 明确是"对抗者"
