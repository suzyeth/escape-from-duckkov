# Step 02: 需求分析 [MODE: EXPLORE]

## 📍 当前位置
步骤 2/6 | 前置：知识库已加载 | 后续：方案设计 | 🧭 探索模式

## 🎯 本步骤目标
深入分析用户需求，判断需求边界、复杂度，决定是否需要模块拆分。

## 📋 执行清单

1. □ **分析用户需求**：理解真实意图，挖掘潜在需求
2. □ **检查歧义**：如有歧义则提问澄清（⚠️ 唯一可暂停的情况）
3. □ **创建需求文件夹** `.GameDev/REQ-XXX/`
4. □ **判断是否需要模块拆分**（见下方触发条件）

## 🔀 模块拆分触发条件

| 条件 | 阈值 |
|------|------|
| 独立子系统数 | ≥ 3个 |
| 验收标准数 | > 15条 |
| 复杂度评估 | 复杂 |
| 上下文预警 | 使用 > 60% |

## 🔀 拆分原则
1. **功能独立**: 每个模块是独立的功能单元
2. **边界清晰**: 模块间的交互接口明确
3. **可独立验收**: 每个模块有独立的验收标准
4. **适当粒度**: 每个模块的验收标准控制在5-8条

**如需拆分**: 创建模块设计卡片（参考 `templates/模块设计卡片模板.md`）

## ⚠️ 本步骤常见陷阱
- 只看字面需求，不挖掘深层意图
- 该拆分的不拆分，导致后续上下文溢出
- 不该拆分的强行拆分，增加不必要的复杂度
