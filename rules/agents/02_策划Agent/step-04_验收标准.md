# Step 04: 验收标准 [MODE: DESIGN]

## 📍 当前位置
步骤 4/6 | 前置：方案设计已完成 | 后续：全局同步 | 📐 设计模式

## 🎯 本步骤目标
编写清晰、可量化、可自动化测试的验收标准。QA Agent将基于此编写自动化测试代码。

## 📋 执行清单

1. □ **编写正常流程验收标准**（核心功能的正向验证）
2. □ **编写边界条件验收标准**（上限、下限、临界值）
3. □ **编写异常情况验收标准**（错误输入、无效状态、并发冲突）
4. □ **检查数量**：简单功能 ≥3条，中等 ≥5条，复杂 ≥8条
5. □ **逐条自检**：每条标准能否直接转化为自动化测试？

## 📝 验收标准格式

```markdown
- [ ] **AC-001**: [具体可测试的条件]
  示例✅: "当玩家拾取物品时，背包物品数量增加1"
  示例❌: "拾取物品体验流畅"（不可测试）

- [ ] **AC-002**: [边界条件]
  示例✅: "当背包已满（100/100）时，拾取物品应失败并提示'背包已满'"

- [ ] **AC-003**: [异常条件]
  示例✅: "当物品ID不存在时，使用操作应返回错误并不扣除物品"
```

## ⚠️ 验收标准质量自检

| 检查项 | 通过？ |
|--------|--------|
| 每条标准是否包含明确的输入条件？ | □ |
| 每条标准是否包含可验证的预期结果？ | □ |
| 是否存在"流畅"、"好用"、"合理"等主观词汇？ | □（应为否） |
| 是否覆盖正常+边界+异常三类场景？ | □ |
| QA能否直接基于此编写测试用例？ | □ |

## ⚠️ 本步骤常见陷阱
- 使用主观描述词汇（"体验好"、"响应快"）
- 只写正常流程，遗漏边界和异常
- 验收标准过少，覆盖不全
- 验收标准之间有重叠或矛盾
