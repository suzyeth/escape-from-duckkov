# 🔄 需求类型与流转路径

> 本文件为开发流水线核心工作流规则，从主rule.md中拆分。
> 触发方式：手动触发（当需要执行开发流水线时加载）
> 版本: v3.8 | 更新: 2026-03-04 | 新增规模自适应增强（XS/S/M标准模式 + L主从模式 + XL深度模式），借鉴BMAD

---

## 需求类型分类

| 类型代码 | 需求类型 | 说明 | 流转路径 |
|----------|----------|------|----------|
| `FEATURE` | 功能开发 | 新功能、新玩法 | 制作人 → 项目管理 → 策划 → [UX] → 主程 → 程序 → QA → **策划(交付)** |
| `FEATURE_UI` | 功能开发(含UI) | 涉及UI界面的新功能 | 制作人 → 项目管理 → 策划 → **UX** → 策划确认 → 主程 → 程序 → QA → **策划(交付)** |
| `OPTIMIZE` | 代码优化 | 性能优化、重构 | 制作人 → 项目管理 → 主程 → 程序 → QA → **策划(交付)** |
| `BUGFIX` | Bug修复 | 修复已知问题 | 制作人 → 项目管理 → 程序 → QA → **策划(交付)** |
| `TEST` | 测试相关 | 增加测试用例 | 制作人 → 项目管理 → QA → **策划(交付)** |
| `DOC` | 文档相关 | 检查/更新**技术**文档 | 制作人 → 项目管理 → 主程 |
| `REVIEW` | 代码审查 | 代码质量检查 | 制作人 → 项目管理 → 主程 → 程序 |
| `CONFIG` | 配置调整 | 参数调整 | 制作人 → 项目管理 → 程序 → QA → **策划(交付)** |
| `RESEARCH` | 方向调研 | 功能调研、竞品分析、待做清单、**策划案梳理** | 制作人 → 策划 |

> **注意**: 除DOC、REVIEW、RESEARCH外，所有需求类型在QA验证通过后都需要流转回策划Agent进行交付（更新全局策划案、待做清单、需求池状态）。

> **注意**: `FEATURE` 类型中，如果策划案涉及UI界面，策划完成后会自动流转给UX Agent进行交互设计，UX设计完成后需要策划确认，确认通过后再流转给主程。

> **🔴 重要：产出物归属原则**: DOC类型仅覆盖**技术类文档**（归主程Agent），**策划类文档**（全局策划案、策划案等）归RESEARCH类型（归策划Agent）。当需求同时包含策划文档和技术文档时，**必须拆分为多个子需求**分别走对应类型的流转路径，禁止合并为单一DOC类型。

> **🔴 重要：自我迭代强制规则**: 当AI被用户纠正错误时（如分类错误、越权、遗漏拆分等），不仅要修正当前任务，**还必须立即对规则文件进行迭代更新**（5 Why分析→定位根因→更新规则文件→记录迭代），确保同类错误不会再次发生。这不是可选步骤，而是强制步骤。

### 项目管理监控机制

```
每个阶段执行时：

│  │                                                          │
│  └─ 阶段结束时：记录实际时长，更新统计                        │
└─────────────────────────────────────────────────────────────┘
```

### Bug 流转机制

- QA发现测试失败 → 生成Bug报告 → 自动流转程序修复 → 重新测试
- 每个Bug最多修复 **3轮**，超过则暂停请求用户介入

## 🚧 质量门禁补充说明

> 门禁总览表和执行规则见主rule，以下为本文件独有的补充说明。

### 关卡1特色：实现泄漏检查

> **实现泄漏** = 策划案中混入了技术实现细节

策划案应聚焦于"**做什么**"而非"**怎么做**"。如果发现策划案中出现了具体类名、数据结构选型、设计模式选择等技术细节，需要清理后才能流转。

### 关卡3特色：验收标准覆盖审计

> **覆盖审计** = 逐条对照策划案的AC-XXX，检查代码是否真正覆盖

不仅检查代码质量，更确保每条验收标准在代码中都有对应的实现和边界处理。

## 🔍 Smart Probe 工作流

```
需求分析完成
    ↓
5维度清晰度评分（用户价值/范围边界/交互方式/数值参数/关联影响）
    ↓
┌──────────────────────────────────────────┐
│  总分 ≥ 20  →  ✅ 直接开始方案设计       │  ← 大多数需求走这条路
│  总分 15-19 →  🟡 AI补充假设，标注后继续  │  ← 小部分需求
│  总分 < 15  →  🔴 暂停，提问(最多3个)     │  ← 极少数需求
└──────────────────────────────────────────┘
    ↓
方案设计
```

## 📋 全局文档维护机制

### 全局策划案 (`全局策划案.md`)

**用途**: 汇总所有功能的策划设计，让任何人都能快速了解整个游戏的功能全貌

**更新时机**:
- 策划 Agent 完成单个功能策划案后，同步更新全局策划案
- 功能被取消或大幅修改时，同步更新

**内容结构**:
- 游戏概述
- 功能模块列表（按系统分类）
- 每个功能的简要说明和链接
- 功能间的关联关系

### 全局技术文档 (`全局技术文档.md`)

**用途**: 汇总所有功能的技术实现，让开发者快速了解整体架构

**更新时机**:
- 主程 Agent 完成单个功能技术设计后，同步更新全局技术文档
- 代码重构或架构调整时，同步更新

**内容结构**:
- 技术架构概览
- 核心系统列表
- 每个系统的技术要点和链接
- 系统间的依赖关系
- 公共组件和工具类

## 📏 规模自适应：各模式详细说明

> 规模评估标准表见主rule，以下为各模式的详细执行说明。

### 标准模式（XS/S/M）

适用于：300行以内代码，不涉及多模块协作

```
完整流程不变：
制作人 → 项目管理 → 策划 → [UX] → 主程 → 程序 → QA → 策划(交付)
                                              ↑
                                         质量门禁1/2/3正常执行
```

- 所有质量门禁正常执行
- 对抗审查正常执行
- 不启用主从架构
- 这是绝大多数需求的默认模式

### 主从模式（L）

适用于：多模块协作的大型需求，300行以上代码

```
在标准流程基础上启用主从Agent架构：

制作人 → 项目管理 → 主策划(拆分模块) → [UX] → 主程(整体架构) → 程序(分模块实现) → 主QA(分模块测试) → 策划(交付)
                       ↓                           ↓                    ↓                    ↓
                    子策划×N                      子任务×N              子程序×N              子QA×N
                       ↓                           ↓                    ↓                    ↓
                    汇总策划案                    汇总技术设计          汇总代码              汇总测试
```

**L规模的特殊执行规则**：
1. 主策划拆分为多个模块，由子策划Agent分别完成设计
2. 主程设计整体架构，为每个模块创建子任务卡片
3. 程序Agent按子任务卡片分别实现各模块代码
4. 主QA规划测试策略，由子QA Agent分别执行各模块测试
5. 主Agent负责汇总和一致性检查
6. 参考：`.ai-rules/rules/agents/主从Agent架构.md`

**触发条件**（满足任一即触发）：
- 制作人评估规模为 L
- 功能包含 3个以上独立模块
- 预估代码超过 300行

### 深度模式增强（XL）

适用于：新系统/跨系统改造

```
在主从模式基础上，增加以下强制环节：
                                                    
制作人 → 项目管理 → 主策划 → 🔍Smart Probe(强制) → [UX] → 主程 → 🏗️架构评审(强制) → 程序 → ⚔️对抗审查(强制) → QA → 📝测试策略文档(强制) → 策划(交付)
```

**XL规模的特殊执行规则**：
1. **🔍 需求深度探测（强制）**：无论清晰度评分多高，XL需求必须执行Smart Probe
2. **🏗️ 架构评审（强制）**：主程完成技术设计后，必须进行架构层面的额外评审
3. **⚔️ 对抗性审查（强制+加强）**：审查问题数量上限提升至 15个，最少发现 5个
4. **📝 测试策略文档（强制）**：QA必须先产出测试策略文档，再编写测试代码
5. 同时启用主从架构（继承L规模的所有规则）

**触发条件**（满足任一即触发）：
- 制作人评估规模为 XL
- 涉及全新系统开发
- 涉及跨系统/跨模块深度改造

### 规模与需求类型的关系

规模评估**独立于**需求类型判断，两者正交组合：

| 需求类型 \ 规模 | XS/S/M | L | XL |
|----------------|--------|---|-----|
| FEATURE | 标准流程 | 主从模式 | 深度模式 |
| FEATURE_UI | 标准流程+UX | 主从模式+UX | 深度模式+UX |
| OPTIMIZE | 标准流程(跳过策划) | 主从模式(跳过策划) | 深度模式(跳过策划) |
| BUGFIX | 标准流程(跳过策划+主程) | 不适用（Bug不会是L规模） | 不适用 |
| CONFIG | 标准流程(跳过策划+主程) | 不适用 | 不适用 |
| TEST | 标准流程(直接QA) | 不适用 | 不适用 |
| DOC | 标准流程(直接主程) | 不适用 | 不适用 |
| REVIEW | 标准流程(主程→程序) | 不适用 | 不适用 |
| RESEARCH | 标准流程(直接策划) | 不适用 | 不适用 |

> **注意**：BUGFIX/CONFIG/TEST/DOC/REVIEW/RESEARCH 通常只有中小规模，不需要L/XL模式。只有 FEATURE/FEATURE_UI/OPTIMIZE 可能触发L/XL模式。
