## 📁 迭代文档体系

```
.GameDev/_Iteration/
├── 知识库.md                # 主知识库（模块知识、技术知识、数据统计）
├── 知识库_策划.md           # 策划职能知识库
├── 知识库_UX.md             # UX职能知识库
├── 知识库_主程.md           # 主程职能知识库
├── 知识库_程序.md           # 程序职能知识库
└── 知识库_QA.md             # QA职能知识库
```

**简化原则：发现问题 → 直接更新对应Agent文件 → 完成**

不再使用独立的迭代数据文件，原因：
- ❌ 多一个文件 = 多一个维护负担
- ❌ 记录了还要读取 = 流程复杂
- ✅ 直接改Agent规则 = 问题不会再发生

> **⭐ 微文件架构下的迭代规则**：
> - 核心Agent已改造为入口文件+步骤文件结构
> - **发现步骤级别的问题** → 直接更新对应的step文件
> - **发现角色级别的问题** → 更新Agent入口文件
> - **发现模板问题** → 更新对应的templates文件
> - 更新范围更小、更精准，避免了以前改大文件的风险

## 🔍 简化迭代机制

### 🔒 不可变规则保护（借鉴ECC Rules Layer — 迭代前必查）

> **🚨 核心原则**：迭代四步法在第②步"定位需更新的规则文件"时，**必须先判断目标规则的层级**。
> 如果目标规则属于 🔴 不可变核心规则（I-001~I-009），**必须停止迭代并向用户报告**。

**迭代四步法增强流程**：

```
① 5 Why根因分析 — 找到根本原因
    │
② 定位需更新的规则文件
    │
    ├── 🔴 目标规则是不可变核心规则（I-001~I-009）？
    │   └── ⛔ 停止迭代 → 向用户报告：
    │       "该问题涉及不可变核心规则 I-XXX，需要用户明确指令才能修改"
    │
    ├── 🟡 目标规则是可迭代流程规则？
    │   └── ✅ 继续执行 → 按标准迭代流程更新
    │
    └── 🟢 目标规则是项目适应规则？
        └── ✅ 直接修改 → 无需走完整迭代流程
    │
③ 立即执行更新（仅限 🟡🟢 层级）
    │
④ 记录迭代日志
```

**规则层级速查表**（迭代时快速判断）：

| 层级 | 标识 | 修改权限 | 判断方法 |
|------|------|---------|---------|
| 🔴 不可变 | `I-001`~`I-009` | ⛔ 需用户明确指令 | 查看 `rule.md` 中 "🔴 不可变核心规则" 章节 |
| 🟡 可迭代 | 流转路径/门禁/防护规则等 | ✅ 走迭代四步法 | 查看 `rule.md` 中 "🟡 可迭代流程规则" 章节 |
| 🟢 项目适应 | 命名/风格/知识库/技能包等 | ✅ 直接修改 | 查看 `rule.md` 中 "🟢 项目适应规则" 章节 |

**常见误触场景**：

| 场景 | 错误做法 | 正确做法 |
|------|---------|---------|
| 觉得质量门禁太严想跳过 | ❌ 迭代修改"去掉门禁" | ✅ 报告用户：涉及I-003，需确认 |
| 觉得.meta检查多余 | ❌ 迭代修改"取消G-001" | ✅ 报告用户：涉及I-001，需确认 |
| 想让主程直接写代码 | ❌ 迭代修改"主程可写代码" | ✅ 报告用户：涉及I-006越权，需确认 |
| 想简化入口文件阅读步骤 | ❌ 迭代修改"跳过阅读" | ✅ 报告用户：涉及I-008，需确认 |

### 实时触发（立即更新）

| 触发条件 | 处理方式 |
|----------|----------|
| 流程无法继续执行 | 立即更新对应的step文件或Agent入口文件 |
| 发现文档描述与实际不符 | 立即修正对应的step/template文件 |
| 发现遗漏的关键步骤 | 立即补充到对应的step文件 |
| 发现更好的实践方式 | 更新到知识库.md或对应step文件 |
| 发现模板不实用 | 直接更新templates目录下的模板文件 |

### 阶段结束触发（阶段反思）

每个 Agent 完成阶段任务后，执行**阶段反思检查**：

```
📋 阶段反思检查清单

□ 本阶段是否遇到文档描述不清的情况？
□ 本阶段是否有步骤可以优化或省略？
□ 本阶段是否发现了新的最佳实践？
□ 本阶段是否有重复性工作可以模板化？
□ 本阶段的输出是否被下游 Agent 顺利接收？
□ 本阶段是否有信息需要补充到全局文档？
□ 本阶段是否有行为偏离人格档案？（对照入口文件中的行为底线逐条检查）
```

### 需求完成触发（🪞 结构化需求级内省，借鉴SuperClaude /sc:reflect）

> **⬆️ 升级说明**：原有的"需求复盘检查清单"已升级为**结构化内省机制**，从"勾选清单"升级为"深度分析+自动归档"。

每个需求完成交付（step-D1）后，**自动执行** `step-D2_需求内省.md`：

**与旧版复盘的差异**：

| 维度 | 旧版复盘 | 新版结构化内省 |
|------|---------|---------------|
| 形式 | 勾选清单（☐ 布尔判断） | 结构化表格（具体数据+分析） |
| 深度 | "是否有返工？"→勾☐ | "返工X次，根因是XXX，改进方案是XXX" |
| 产出 | 无单独产出 | 生成 `07_内省报告.md` |
| 经验归档 | 手动判断 | 按规则自动归档到对应文件 |
| 时间 | 无限制 | 严格限时5分钟 |
| 指标 | 无量化指标 | 4项效率指标对比历史均值 |

**执行流程概要**：
```
step-D1交付完成 → 自动进入 step-D2
    │
    ├── 1. 收集基础数据（从frontmatter读取时长/返工/门禁数据）
    ├── 2. 识别成功模式（≥ 1个）
    ├── 3. 识别失败模式（含简要5Why）
    ├── 4. 提炼经验（通用/项目特定）
    ├── 5. 计算效率指标（时长准确率/返工率/门禁通过率/审查发现数）
    ├── 6. 经验自动归档（🔴立即/🟡延后/🟢可选）
    └── 7. 生成内省报告（.GameDev/{name}/07_内省报告.md）
```

> 详细执行清单见：`02_策划Agent/step-D2_需求内省.md`
> 报告模板见：`02_策划Agent/templates/需求内省模板.md`

**精简版触发条件**：当需求返工次数=0、Bug修复≤ 1轮、门禁全部一次通过、未被用户纠正时，可执行精简版内省。

### 周期性触发（🪞 结构化周期级内省，借鉴SuperClaude /sc:reflect）

每完成 **3个需求** 或 **累计发现5个改进点** 时，或用户主动输入 `/gd:reflect`，执行**周期级结构化内省**。

> **💡 与审计的关系**：周期级内省关注「执行效果」（做得好不好），定期审计关注「规则健康度」（规则体系是否退化）。两者互补，审计每5个需求触发，内省每3个需求触发，交替执行覆盖更全面。

**周期级内省与需求级内省的关系**：

| 内省级别 | 触发时机 | 分析范围 | 深度 | 产出 |
|---------|---------|---------|------|------|
| **阶段级** | 每个Agent阶段流转前 | 当前阶段 | 快速检查 | 无单独产出（已有） |
| **需求级** | 每个需求交付后（step-D2） | 本次需求全流程 | 中度分析 | `07_内省报告.md` |
| **周期级** | 每3个需求或`/gd:reflect` | 最近N个需求 | 深度分析 | 周期内省报告 |

**周期级内省执行流程**：
```
触发周期内省（/gd:reflect 或自动触发）
    │
    ├── 1. 读取需求池，汇总最近已完成需求的数据
    ├── 2. 读取各需求的内省报告（07_内省报告.md）
    ├── 3. 计算核心指标均值和趋势（对比目标值）
    ├── 4. 识别反复出现的成功/失败模式
    │   └── 特别关注：出现≥2次且未修复的失败模式
    ├── 5. 评估规则文件健康度（修改频率+效果）
    ├── 6. 统计知识库增长情况
    ├── 7. 使用 `02_策划Agent/templates/周期内省模板.md` 生成报告
    └── 8. 输出行动清单（按优先级排序）
```

> 报告模板见：`02_策划Agent/templates/周期内省模板.md`

## 🔬 定期审计（借鉴ECC /harness-audit）

> **核心理念**：规则体系是活的系统，需要**定期体检**发现退化、遗漏、不一致等健康问题。
> 从"出了问题才修规则"升级为"定期巡检，防患于未然"。

### 审计触发规则

| 触发方式 | 触发条件 | 说明 |
|---------|---------|------|
| 🔄 自动触发 | 每完成**5个需求** | 在第5/10/15...个需求交付后自动执行（在需求级内省之后） |
| 🔄 自动触发 | 周期级内省发现≥3个异常 | 内省中发现规则问题较多时，自动追加审计 |
| 🖐️ 手动触发 | 用户输入 `/gd:audit` | 随时可执行全面审计 |
| 🖐️ 手动触发 | 用户输入 `/gd:audit 知识库` | 只审计指定维度（部分审计） |

### 审计执行约束

| 约束项 | 值 | 说明 |
|--------|---|------|
| **时间上限** | 10分钟 | 超时则停止，输出已完成的检查结果 |
| **输出原则** | 只输出异常项 | 正常项不展示，减少信息噪音 |
| **结果分级** | 🔴🟡🟢 三级 | 🔴需立即修复 / 🟡建议关注 / 🟢信息提示 |
| **行动清单** | 自动生成 | 审计结束后按优先级输出可执行的修复建议 |

### 6维度审计检查清单

#### 📏 维度1：规则完整性

```
□ rule.md 版本号与最近更新记录是否一致
□ 所有子规则文件是否在 rule.md 索引中
□ 不可变核心规则（I-001~I-009）是否被意外修改
□ 禁止事项列表与防护规则（rule_guards.md）是否一致
□ 规则分层体系三层是否完整（🔴🟡🟢）
□ 迭代四步法的不可变规则保护是否生效
```

异常示例：
- 🔴 `rule_guards.md 中的 G-001 定义与 rule.md 中 I-001 描述不一致`
- 🟡 `rule_commands.md 版本号(v1.2)与 rule.md 中记载的更新历史不匹配`

#### 🤖 维度2：Agent定义一致性

```
□ 所有Agent入口文件是否存在（8个Agent，对照rule.md Agent列表）
□ Agent职责速查表（rule.md）与各入口文件的职责描述是否一致
□ 工具权限声明（🔐）是否已添加到所有Agent入口文件
□ 行为模式标签 [MODE: xxx] 是否已标注到所有step文件标题
□ 每个Agent的步骤索引（入口文件）与实际step文件是否对应
□ Agent人格档案是否完整（名称、沟通风格、决策原则、行为底线）
```

异常示例：
- 🔴 `04_程序Agent.md 中的步骤索引列出了 step-06，但该文件不存在`
- 🟡 `07_UXAgent.md 缺少 🔐 工具权限声明章节`

#### 📚 维度3：知识库健康度

```
□ 各知识库文件是否存在（主知识库 + 5个职能知识库）
□ 知识库是否有最近更新（最近5个需求内是否有新增经验）
□ 新增经验条目是否使用增强格式（置信度+触发关键词+应用方式）
□ 是否有重复或矛盾的经验条目（同一功能不同结论）
□ 是否有标记"待复查"或"已废弃"的过期条目需要清理
□ 🟢高置信度经验是否有实际自动应用的记录
□ 旧版无结构化经验条目的比例（目标：逐步减少）
```

异常示例：
- 🟡 `知识库_程序.md 最近10个需求无新增经验，可能存在经验沉淀遗漏`
- 🟢 `知识库_策划.md 中仍有12条旧版无结构化经验条目，建议逐步升级`

#### 🛡️ 维度4：防护规则有效性

```
□ rule_guards.md 中的防护规则列表是否完整（事前G-001~G-010 + 事后G-101~G-107）
□ 是否有新增的高频问题需要添加防护规则（检查最近需求的内省报告中的失败模式）
□ 防护规则总数是否在合理范围（≤20条，避免拖慢效率）
□ 是否有防护规则长期未触发（可能已过时，标记待复查）
□ 事前阻断规则（🔴）是否在编码步骤中被正确执行
```

异常示例：
- 🔴 `最近3个需求的内省报告均提到"序列化兼容问题"，但无对应防护规则`
- 🟢 `G-010（修改ProjectSettings警告）最近10个需求未触发，可能已过时`

#### 📊 维度5：指标达标情况

```
□ 返工率是否 < 20%（统计最近5个需求）
□ 平均Bug修复轮数是否 < 1.5
□ 时长预估准确率是否 > 80%
□ 问题重复率是否 < 10%
□ 质量门禁一次通过率是否 > 70%
□ 对抗审查发现率是否 ≥ 3个/次
□ 状态追踪完整率是否 = 100%
□ Smart Probe探测精准度是否 > 85%
```

异常示例：
- 🔴 `返工率 35%（目标 <20%），最近5个需求中有2个返工超过2次`
- 🟡 `质量门禁一次通过率 60%（目标 >70%），门禁2（主程→程序）频繁打回`

#### 📋 维度6：文档同步性

```
□ rule.md 中的Agent职责表与各Agent入口文件是否一致
□ rule_commands.md 中的命令列表是否与实际可用命令一致
□ 模板文件（templates/）是否为最新版本
□ rule.md 的"子规则文件索引"是否包含所有实际存在的子规则文件
□ 各step文件中引用的其他文件路径是否正确
□ 全局策划案/全局技术文档与最新需求的内容是否同步
```

异常示例：
- 🔴 `rule.md 中 Agent 列表有8个，但实际只有7个入口文件（缺少美术Agent）`
- 🟡 `策划Agent step-03 中引用的模板路径已变更，需同步更新`

│  └── 周期级：每3个需求（/gd:reflect，深度分析）                │
│                                                                │

│  ├── 定期审计：每5个需求（/gd:audit，全面体检）                │
│  ├── 追加审计：周期内省发现≥3个异常时                          │
│  └── 手动审计：用户随时触发                                    │
│                                                                │
│  🛡️ 防护（看实时操作）                                         │
│  ├── 事前防护：每次文件操作前自动检查（G-001~G-010）           │
│  └── 事后检查：编码完成后自动检查（G-101~G-107）               │
│                                                                │
│  📊 触发时间线（以10个需求为例）：                              │
│  REQ-1 ─ REQ-2 ─ REQ-3[内省] ─ REQ-4 ─ REQ-5[内省+审计]      │
│  REQ-6[内省] ─ REQ-7 ─ REQ-8 ─ REQ-9[内省] ─ REQ-10[内省+审计]│

## 🧠 直觉学习增强（借鉴ECC Instincts System）

> **核心理念**：不仅积累经验，更让经验在合适的时候**自动发挥作用**。
> 从"经验写了→放那了→偶尔翻到"升级为"经验写了→自动匹配→主动提示/自动应用"。

### 经验条目增强格式

知识库中的经验条目从简单文本升级为结构化格式：

```markdown
#### [经验标题]
- **置信度**: 🟢高 / 🟡中 / 🔵低
- **触发关键词**: [关键词1, 关键词2, ...]
- **适用Agent**: [程序/主程/QA/策划/全部]
- **来源**: [REQ-XXX / 用户教授(/gd:learn) / 多次验证 / 内省发现]
- **内容**: [具体经验描述]
- **应用方式**: 🔴自动应用 / 🟡提示应用 / 🟢参考
```

**格式说明**：
| 字段 | 必填 | 说明 |
|------|------|------|
| 经验标题 | ✅ | 简明标题，便于快速识别 |
| 置信度 | ✅ | 🟢高=多次验证或用户教授，🟡中=1-2次验证，🔵低=首次发现 |
| 触发关键词 | ✅ | 用于自动匹配，从经验内容中提取核心词 |
| 适用Agent | ✅ | 标注哪些Agent需要关注此经验 |
| 来源 | ✅ | 追溯经验的产生场景 |
| 内容 | ✅ | 具体的经验描述和做法 |
| 应用方式 | ✅ | 根据置信度自动确定（见下方映射规则） |

**置信度→应用方式映射**：

| 置信度 | 应用方式 | 行为 |
|--------|---------|------|
| 🟢 高 | 🔴 自动应用 | Agent遇到匹配场景时直接使用，输出标注 `"💡[经验] 自动应用：XXX"` |
| 🟡 中 | 🟡 提示应用 | Agent遇到匹配场景时输出建议 `"💡[建议] 根据历史经验，建议：XXX"` |
| 🔵 低 | 🟢 参考 | 不主动提示，仅在精准加载时可见 |

**置信度晋升规则**：
| 场景 | 置信度变化 |
|------|----------|
| 经验首次从内省中发现 | 🔵低 |
| 同一经验被第2个需求验证 | 🔵低 → 🟡中 |
| 同一经验被第3+个需求验证 | 🟡中 → 🟢高 |
| 用户通过 `/gd:learn` 主动教授 | 直接 🟢高 |
| 用户纠正错误后提炼的经验 | 直接 🟡中（严重错误→🟢高） |
| 经验被证明不正确 | ⬇️ 降级或标记废弃 |

### 经验自动匹配与应用机制

```
Agent进入某步骤 → 读取对应知识库
    │
    ├── 从当前任务中提取关键词（功能名、模块名、技术栈等）
    │
    ├── 扫描知识库中的经验条目
    │   │
    │   ├── 匹配"触发关键词"？
    │   │   ├── 匹配 + 置信度🟢高 → 🔴 自动应用
    │   │   │   └── 在输出中标注："💡[经验] 自动应用：XXX"
    │   │   │   └── 直接按经验内容执行
    │   │   │
    │   │   ├── 匹配 + 置信度🟡中 → 🟡 提示应用
    │   │   │   └── 在输出中标注："💡[建议] 根据历史经验，建议：XXX"
    │   │   │   └── Agent自行判断是否采纳
    │   │   │
    │   │   └── 匹配 + 置信度🔵低 → 🟢 仅参考
    │   │       └── 不主动提示（已读取，心中有数即可）
    │   │
    │   └── 不匹配 → 跳过
    │
    └── 继续执行步骤
```

**自动应用示例**：

```
场景：程序Agent开始编码"商品列表排序功能"

1. 提取关键词：招募、面板、排序、UI
2. 扫描知识库_程序.md
3. 匹配到经验条目：
   #### 商品列表面板列表需要先按品质降序再按等级降序
   - 置信度: 🟢高
   - 触发关键词: [招募, 面板, 排序, 列表]
   - 来源: REQ-005 / 用户纠正
   - 内容: 商品列表面板的物品列表排序规则为先按品质降序，再按等级降序，不能只用单一排序
   - 应用方式: 🔴自动应用

4. 输出："💡[经验] 自动应用：商品列表面板列表需按品质降序→等级降序排序（来源：REQ-005）"
5. 编码时直接采用此排序规则
```

### `/gd:learn` 命令（用户主动教授知识）

> 借鉴 ECC 的 `/learn` 命令，用户可随时主动教授知识，无需等待需求流程。

**命令格式**：
```
/gd:learn "经验内容"
/gd:learn -agent 程序 "所有Manager类都必须继承BaseManager"
/gd:learn -kb 主 "商品列表面板使用特殊排序规则：品质→等级"
```

**执行流程**：
```
用户输入 /gd:learn "所有Manager类都必须继承BaseManager，不要直接用MonoBehaviour"
    │
    ├── 1. 解析经验内容
    ├── 2. 自动提取触发关键词：[Manager, BaseManager, MonoBehaviour, 继承]
    ├── 3. 自动判断适用Agent：程序（涉及代码编写）
    ├── 4. 设置置信度：🟢高（用户主动教授 = 高置信度）
    ├── 5. 确定写入目标：知识库_程序.md
    ├── 6. 按增强格式写入知识库：
    │   #### Manager类必须继承BaseManager
    │   - 置信度: 🟢高
    │   - 触发关键词: [Manager, BaseManager, MonoBehaviour, 继承]
    │   - 适用Agent: 程序
    │   - 来源: 用户教授(/gd:learn)
    │   - 内容: 所有Manager类都必须继承BaseManager，不要直接用MonoBehaviour
    │   - 应用方式: 🔴自动应用
    │
    └── 7. 输出确认：
        "✅ 已学习：'Manager类必须继承BaseManager'"
        "   → 写入：知识库_程序.md"
        "   → 置信度：🟢高（自动应用）"
        "   → 触发关键词：Manager, BaseManager, MonoBehaviour, 继承"
```

**可选参数**：
| 参数 | 说明 | 默认值 |
|------|------|--------|
| `-agent` | 指定适用Agent | 自动判断 |
| `-kb` | 指定写入知识库（主/策划/程序/主程/QA/UX） | 自动判断 |
| `-confidence` | 指定置信度（高/中/低） | 🟢高 |

### 经验条目与现有知识库的兼容

> **向后兼容**：现有知识库中的非结构化经验条目继续有效，视为 🔵低置信度，不触发自动应用。
> 随着新需求的积累，逐步将旧条目升级为增强格式。

| 经验类型 | 格式 | 置信度 | 自动应用 |
|---------|------|--------|----------|
| 旧版简单文本 | 无结构 | 🔵低（默认） | ❌ 不触发 |
| 新版增强格式 | 结构化 | 按标注 | ✅ 按置信度触发 |
| `/gd:learn` 写入 | 结构化 | 🟢高（默认） | ✅ 自动应用 |

### 经验的生命周期管理

```
经验产生 → 写入知识库（带置信度）
    │
    ├── 后续需求验证有效 → 置信度晋升（🔵→🟡→🟢）
    ├── 后续需求发现无效 → 置信度降级或标记废弃
    ├── 用户明确否定 → 立即标记废弃
    └── 长期未触发（>10个需求） → 标记"待复查"（周期内省时清理）
```

**废弃标记格式**：
```markdown
#### ~~[经验标题]~~
- **状态**: ❌ 已废弃（原因：XXX）
- **废弃时间**: 2026-XX-XX
```

### 🗜️ 知识库精准加载规则（借鉴SuperClaude Token优化）

> **核心原则**：不要全量加载知识库，按需求关键词匹配相关章节，减少无关token消耗。

**精准加载流程**：
```
步骤1：从用户需求中提取关键词（功能名、模块名、技术栈等）
    │
步骤2：读取知识库的章节标题列表（只读目录，不读正文）
    │
步骤3：用关键词匹配相关章节标题
    │
    ├── 有匹配 → 只加载匹配的章节内容
    ├── 无匹配 → 标记"知识库无相关内容"，跳过
    └── 不确定 → 全量加载（宁多不漏）
```

**各知识库的加载策略**：

| 知识库 | 加载策略 | 原因 |
|--------|---------|------|
| 主知识库 | 🗜️ **精准加载** | 内容多，包含所有模块知识，必须精准匹配 |
| 职能知识库 | 📖 **全量读取** | 内容简短，且都是本职能相关经验 |
| 全局策划案 | 🗜️ **精准加载** | 内容多，只需读取与本需求关联的模块 |
| 全局技术文档 | 🗜️ **精准加载** | 内容多，只需读取与本需求关联的技术架构 |
| 成功/失败模式库 | 👁️ **快速扫描** | 只看标题，发现相关模式再深入阅读 |

**精准加载示例**：
```
需求："修复商品列表面板的排序问题"
→ 关键词：招募、面板、排序、UI
→ 主知识库：加载 "招募系统" 章节 ✅，跳过 "对话系统""战斗系统" 等无关章节 ⏭️
→ 知识库_程序：全量加载（短小且都相关）
→ 全局技术文档：加载 "UI架构" 章节 ✅，跳过 "网络架构" 等无关章节 ⏭️
```

### ↩️ 安全回退机制

| 场景 | 回退策略 |
|------|----------|
| 不确定关键词是否匹配 | 全量加载（宁多不漏） |
| 精准加载后发现缺少信息 | 补加读取遗漏的章节 |
| 首次处理某个模块 | 全量加载（无历史经验可参考） |
| 知识库文件很小（<2KB） | 直接全量加载 |

> **核心原则**：每个step文件中明确标注"本步骤只需哪些文件的哪些部分"，避免读取无关内容消耗token。

## 🔄 rule.md 自身迭代检查

> **💡 与审计的关系**：以下检查清单是 `/gd:audit` 审计命令的**检查项来源**之一。
> 手动检查时使用此清单，自动审计时由 `/gd:audit` 按6维度结构化执行。

本文件体系作为全局工作流的核心，需要特别关注以下迭代点：

```
📋 rule.md 自身迭代检查清单

【流程定义】→ 对应审计维度1（规则完整性）
□ 流程图是否准确反映实际执行？
□ 流程步骤描述是否清晰？
□ 流程分支条件是否完整？
□ 异常处理流程是否定义？

【Agent定义】→ 对应审计维度2（Agent定义一致性）
□ Agent职责描述是否准确？
□ Agent间的关系是否清晰？
□ 主从架构描述是否完整？
□ 新增/删除Agent是否需要更新？

□ 快速命令是否完整？

□ 更新记录是否完整？
```

### 📍 经验归档规则（含增强格式要求）

| 经验类型 | 更新位置 | 格式要求 | 示例 |
|----------|----------|---------|------|
| **本项目强相关** | 对应职能知识库 `知识库_XXX.md` | ✅ 必须使用增强格式 | 对话系统的特殊处理、本项目的存档格式、特定UI结构 |
| **其他项目通用** | 对应 `Agent.md` 的最佳实践/常见陷阱章节 | ❌ 无需增强格式（规则文件非知识库） | 验收标准量化、空值检查、边界优先测试 |

**判断标准：**
- 如果换一个项目这个经验还适用 → **通用经验** → 写入对应Agent.md（无需增强格式）
- 如果只有本项目才会遇到这个问题 → **项目经验** → 写入对应职能知识库（**必须使用增强格式**）

**增强格式归档示例**：
```markdown
#### 商品列表面板物品列表排序规则
- **置信度**: 🟢高
- **触发关键词**: [招募, 面板, 排序, 物品列表]
- **适用Agent**: 程序
- **来源**: REQ-005 / 用户纠正
- **内容**: 商品列表面板的物品列表排序必须先按品质降序，再按等级降序，不能只用单一排序字段
- **应用方式**: 🔴自动应用
```
